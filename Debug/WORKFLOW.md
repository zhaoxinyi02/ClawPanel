# ClawPanel 自动处理流程

## 固定循环

1. 编辑前先检查 GitHub 仓库是否有新提交，优先执行 `git fetch origin`、`git status -sb`、必要时查看与 `origin/main` 的差异。
2. 读取 `Result/` 下的项目文档，作为当前排查和修改的上下文基础。
3. 从 `issue.md` 或仓库 issue 中识别尚未解决的问题，优先处理影响安装、更新、核心流程的问题。
4. 每修复一个 issue，都在 `Debug/` 下新增一份调试记录，包含问题、定位、修复方案、验证建议。
5. 完成修改后，先做本地验证；若环境缺少依赖，则明确记录无法验证的原因。
6. 每解决一个 issue 就单独提交一次，提交信息聚焦该 issue 的修复目的。
7. 用户要求同步远端时，推送提交；随后继续进入下一轮循环。

## 当前约定

- `Result/` 作为文档上下文来源。
- `Debug/` 作为每次排查记录目录。
- 保留用户现有未跟踪文件，不主动改动无关内容。
